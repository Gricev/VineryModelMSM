# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Что это

VineryMSM — потоковый сбор данных по ряду винограда с **двух камер**, движущихся
на тележке вдоль ряда. Текущее состояние — рабочий каркас пайплайна: БД, синхронизатор
и локализация полностью реализованы; модели компьютерного зрения (`bush_tracker`,
`canopy_analyzer`) — заглушки-интерфейсы под подключение YOLO/классификаторов.
Документы `DATASET.md` и `DB_SCHEMA.md` — первоисточник по иерархии данных и схеме БД.

Зависимости рантайма (`opencv-python`, `numpy`) нужны только для реального видео;
БД и синхронизатор работают на стандартной библиотеке, поэтому тесты идут без них.
`vinery/pipeline/stream.py` импортируется и без установленного `cv2` (ленивая проверка).

## Архитектура

Поток данных одного прохода тележки (`main.run_pass`):

```
cam1 (ствол лозы)  ──BushTracker.detect──▶ BushEvent ──┐
                                                        ├─▶ Synchronizer ─▶ Database
cam2 (лист/гроздь) ─CanopyAnalyzer.analyze▶ CanopyResult┘   (запись observation)
        ▲                                                        ▲
        └─ оба потока сливаются по времени (merge_streams)       └─ ConstantSpeed/Odometry/Gps Localizer
```

Две камеры дают разные срезы одного **куста** (`bush`) — это связующая сущность,
а не камера. cam1 = «личность» куста (track_id ствола, 1 ствол = 1 куст) + болезнь
лозы; cam2 = признаки (сорт, болезнь листа, стадия соцветия, зрелость/болезнь грозди).

### Ключевой инвариант: идентичность куста = физический якорь, не порядок детекций

Это центральная идея всего проекта (см. `localization.py`, `synchronizer._resolve_anchor`,
`Database.get_or_create_bush`, тест `tests/test_anchoring.py`). Куст определяется
**слотом вдоль ряда**, а не порядком появления в кадре. Поэтому пропуск/задвоение куста
камерой не сдвигает нумерацию остальных, и один физический куст получает один и тот же
`bush_code` (`V1-R03-B017`) в любом проходе. Слот резолвится по приоритету:

1. `BushEvent.marker_code` — метка на шпалере (QR/ArUco/RFID), самый надёжный якорь;
2. координата от `Localizer.localize(t).position_m` → `slot = round(pos / bush_spacing_m) + 1`;
3. порядковый счётчик-fallback (**ненадёжно**, нумерация «сползёт» — не для продакшна).

При добавлении/изменении логики нумерации не ломайте этот инвариант — тест `test_anchoring`
именно его и стережёт. `get_or_create_bush` резолвит куст по `(row_id, position_in_row)`,
так что повторный проезд по тому же слоту переиспользует ту же запись `bush`.

### Synchronizer: склейка двух потоков в наблюдение

`Synchronizer` (`pipeline/synchronizer.py`) держит **одно открытое наблюдение** за раз:
- `on_bush(ev)` с тем же `track_id` продлевает текущее наблюдение; смена `track_id` →
  `_flush()` предыдущего и открытие нового;
- `on_canopy(res)` цепляет признаки cam2 к текущему открытому кусту (если куст ещё не
  открыт камерой 1 — данные cam2 отбрасываются);
- `finish()` дописывает последнее наблюдение в конце прохода.

`_flush()` агрегирует множество кадров в **одно решение по кусту**: классы — голосование
по суммарной уверенности (`_vote`), числа (severity, ripeness) — среднее. Итог = один
`observation` на куст на проход (схема: `UNIQUE(pass_id, bush_id, t_start)`).

### Database: схема в SQL-файле, запись через record-dict

`vinery/db/schema.sql` исполняется при создании `Database` (idempotent, `CREATE IF NOT EXISTS`).
Контракт между синхронизатором и БД — словарь `record` в `Database.write_observation`
(его форма документирована в докстринге метода); поля совпадают с `meta.json` из `DATASET.md`,
чтобы датасет и БД пересобирались один из другого. Меняя поля наблюдения, правьте
**оба места**: `schema.sql` и `write_observation` (+ соответствующую сборку record в `_flush`).

Важно: **болезнь лозы (`vine_health`, cam1)** и **болезнь листа (`leaf_health`, cam2)** —
разные сущности (многолетняя древесина vs сезонные листовые болезни), отдельные таблицы.

## Текущий незавершённый хвост

`main.run_pass` сливает два потока **наивным ручным циклом** (`f1.t <= f2.t`), хотя
`pipeline/stream_sync.merge_streams(cam1, cam2, cam2_offset_s)` — честное слияние по
выровненному времени (учитывает лаг между разнесёнными камерами и разный FPS) — уже
написан и импортирован в `main.py`, но ещё не подключён. Следующий шаг — заменить цикл
в `main.py` на `merge_streams(...)` и пробросить `cam2_offset_s` в `run_pass`.

## Гайд по датасету (при работе с обучением)

Сплиты train/val/test режутся **по `bush_code` (лучше по `vine_row`)**, не по кадру:
кадры одного куста почти идентичны, кадр-уровневый сплит даёт утечку и завышенные метрики.
См. `DATASET.md` §«split по кусту».
