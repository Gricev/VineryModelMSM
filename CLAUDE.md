# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Что это

VineryMSM — потоковый сбор данных по ряду винограда с **двух камер**, движущихся
на тележке вдоль ряда. Текущее состояние — рабочий каркас пайплайна: БД, синхронизатор
и локализация полностью реализованы; модели компьютерного зрения (`bush_tracker`,
`canopy_analyzer`) — заглушки-интерфейсы под подключение YOLO/классификаторов.
Документы `DATASET.md` и `DB_SCHEMA.md` — первоисточник по иерархии данных и схеме БД.

# Точка входа (сейчас только печатает подсказку; запуск прохода — через run_pass())

Зависимости рантайма (`opencv-python`, `numpy`) нужны только для реального видео;
БД и синхронизатор работают на стандартной библиотеке, поэтому тесты идут без них.
`vinery/pipeline/stream.py` импортируется и без установленного `cv2` (ленивая проверка).

## Архитектура

Поток данных одного прохода тележки (`main.run_pass`):

```
cam1 (ствол лозы)  ──BushTracker.detect──▶ BushEvent ──┐
                                                        ├─▶ Synchronizer ─▶ Database
cam2 (лист/гроздь) ─CanopyAnalyzer.analyze▶ CanopyResult┘   (запись observation)
        ▲                                                        ▲
        └─ оба потока сливаются по времени (merge_streams)       └─ ConstantSpeed/Odometry/Gps Localizer
```

Две камеры дают разные срезы одного **куста** (`bush`) — это связующая сущность,
а не камера. cam1 = «личность» куста (track_id ствола, 1 ствол = 1 куст) + болезнь
лозы; cam2 = признаки (сорт, болезнь листа, стадия соцветия, зрелость/болезнь грозди).

### Ключевой инвариант: идентичность куста = физический якорь, не порядок детекций

Это центральная идея всего проекта (см. `localization.py`, `synchronizer._resolve_anchor`,
`Database.get_or_create_bush`, тест `tests/test_anchoring.py`). Куст определяется
**слотом вдоль ряда**, а не порядком появления в кадре. Поэтому пропуск/задвоение куста
камерой не сдвигает нумерацию остальных, и один физический куст получает один и тот же
`bush_code` (`V1-R03-B017`) в любом проходе. Слот резолвится по приоритету:

1. `BushEvent.marker_code` — метка на шпалере (QR/ArUco/RFID), самый надёжный якорь;
2. координата от `Localizer.localize(t).position_m` → `slot = round(pos / bush_spacing_m) + 1`;
3. порядковый счётчик-fallback (**ненадёжно**, нумерация «сползёт» — не для продакшна).

При добавлении/изменении логики нумерации не ломайте этот инвариант — тест `test_anchoring`
именно его и стережёт. `get_or_create_bush` резолвит куст по `(row_id, position_in_row)`,
так что повторный проезд по тому же слоту переиспользует ту же запись `bush`.

### Synchronizer: склейка двух потоков в наблюдение

`Synchronizer` (`pipeline/synchronizer.py`) держит **одно открытое наблюдение** за раз:
- `on_bush(ev)` с тем же `track_id` продлевает текущее наблюдение; смена `track_id` →
  `_flush()` предыдущего и открытие нового;
- `on_canopy(res)` цепляет признаки cam2 к текущему открытому кусту (если куст ещё не
  открыт камерой 1 — данные cam2 отбрасываются);
- `finish()` дописывает последнее наблюдение в конце прохода.

`_flush()` агрегирует множество кадров в **одно решение по кусту**: классы — голосование
по суммарной уверенности (`_vote`), числа (severity, ripeness) — среднее. Итог = один
`observation` на куст на проход (схема: `UNIQUE(pass_id, bush_id, t_start)`).

### Database: схема в SQL-файле, запись через record-dict

`vinery/db/schema.sql` исполняется при создании `Database` (idempotent, `CREATE IF NOT EXISTS`).
Контракт между синхронизатором и БД — словарь `record` в `Database.write_observation`
(его форма документирована в докстринге метода); поля совпадают с `meta.json` из `DATASET.md`,
чтобы датасет и БД пересобирались один из другого. Меняя поля наблюдения, правьте
**оба места**: `schema.sql` и `write_observation` (+ соответствующую сборку record в `_flush`).

Важно: **болезнь лозы (`vine_health`, cam1)** и **болезнь листа (`leaf_health`, cam2)** —
разные сущности (многолетняя древесина vs сезонные листовые болезни), отдельные таблицы.

## Текущий незавершённый хвост

`main.run_pass` сливает два потока **наивным ручным циклом** (`f1.t <= f2.t`), хотя
`pipeline/stream_sync.merge_streams(cam1, cam2, cam2_offset_s)` — честное слияние по
выровненному времени (учитывает лаг между разнесёнными камерами и разный FPS) — уже
написан и импортирован в `main.py`, но ещё не подключён. Следующий шаг — заменить цикл
в `main.py` на `merge_streams(...)` и пробросить `cam2_offset_s` в `run_pass`.

## Обучение (камера 1: детекция ствола лозы)

Пакет `vinery/training` — поэтапный каркас под задачу cam1. Тяжёлые зависимости
(`cv2`, `torch`, `sam2`, `ultralytics`, `roboflow`) импортируются **лениво**, поэтому
тесты конвертации и сборки идут на чистой stdlib. Этапы (см. докстринг `training/__init__.py`):

```
synthetic ─┐
ingest extract (реальное видео -> кадры) ─┤
sam2_prelabel (маска SAM2 -> YOLO-бокс, черновик) ─┴─▶ [разметка в CVAT/Roboflow]
        │                                                        │
        │ roboflow_upload (заливка кадров через API)             ▼
        └──────────────────────────────────────────▶ ingest import (YOLO-экспорт -> датасет)
                                                                 ▼
                                                yolo_dataset (split-by-row) ─▶ train_trunk
```

- **`ingest`** — `extract` режет mp4 проезда на кадры в стейджинг `dataset/to_label/vine_trunk/`
  (имена `V1-R03__pass_01__f0007`, бирка = РЯД, т.к. точный куст на сыром видео неизвестен);
  `import` принимает YOLO-экспорт обратно в раскладку задачи и дописывает `bush_index.csv`.
  `stage-images` — отдельные фото в стейджинг с биркой ряда без cv2. Команды cam2
  (`canopy-extract`, `canopy-stage-images`) живут здесь же — см. раздел камеры 2.
- **`sam2_prelabel`** — авто-ПРЕДразметка: SAM2 даёт маску, `mask_to_yolo_bbox` → бокс. Это
  ЧЕРНОВИК под обязательную ручную проверку. Для фото — режим `images` (промпт на кадр:
  `--prompts` JSON или `--interactive`); для видео в проде рекомендуется SAM2-трекер CVAT.
- **`roboflow_upload`** — заливка папки кадров в Roboflow через API (обход лимита веб-UI;
  zip API не принимает — грузятся одиночные файлы из `--dir`). **Возобновляемо**: успешные
  имена пишутся в `.uploaded.txt` (flush после каждого), повторный запуск пропускает залитое
  и доливает остаток; `--no-resume` — залить всё заново.

При смене формата стейджинга/имён правьте согласованно: `frame_name`/`row_bush_code`
и `canopy_frame_name` (ingest), `_photo_name` (sam2_prelabel),
`_canopy_photo_name` (sam2_canopy) и `IMAGE_EXTS` — от них зависит и split, и приём.

## Обучение (камера 2: лист / соцветие / гроздь)

cam2 — **отдельная дорожка** от cam1: своя таксономия и геометрия, но раскладку каталогов
и анти-утечку (split по ряду) делит с cam1. Первоисточник разметки — **`CANOPY_LABELS.md`**
(контракт «что/какой геометрией/какими кодами»), коды полей совпадают со `schema.sql` и
`CanopyResult` (`pipeline/models.py`), чтобы датасет и БД пересобирались один из другого.

Ключевые решения спека (см. `CANOPY_LABELS.md`):
- **Орган = фенофаза BBCH, а не отдельные классы-конкуренты.** Соцветие и гроздь — один
  репродуктивный орган в разных фазах, в кадре не сосуществуют. **Партия фото = одна
  фенофаза** (диктуется датой съёмки).
- **Геометрия — маски (YOLO-seg), два слоя:** маска ОРГАНА (`leaf`/`inflorescence`/`cluster`)
  + маска ПОРАЖЕНИЯ. `severity = площадь(поражение)/площадь(орган)` считается на приёме из
  масок, а не на глаз. `healthy` размечать обязательно (иначе модель учит «орган = болезнь»).
- **Болезнь соцветия пишется в `cluster_health`** = здоровье репрод. органа на обеих
  фенофазах; отдельной сущности нет. Фенофаза восстанавливается по тому, заполнено
  `inflorescence_stage` или `cluster_ripeness`. (Поэтому `schema.sql`/`models.py` для
  `cluster_health` переподписаны «репрод. орган: соцветие+гроздь», severity = доля площади.)
- **Сорт (`variety`) НЕ размечается покадрово** — берётся из ряда (`vine_row.variety_id`,
  1 сорт/ряд). Поле `CanopyResult.variety` оставлено как опц. кросс-проверка (в пилот не входит).

Стейджинг (реализован в `ingest.py`, общее ядро декодирования с cam1):
- `canopy-extract video.mp4 --row 3 --phase flowering --pass 1` → кадры в
  `dataset/to_label/canopy/<phase>/`, имя `V1-R03__<phase>__pass_01__f0007`
  (`<phase>` ∈ `flowering`/`cluster`/`leaf`). **Ряд — 1-й сегмент** → `bush_code_from_name`
  и split-by-row работают без правок; **фенофаза — 2-й сегмент** → восстановима
  `canopy_phase_from_name`, маршрутизирует кадр в задачу разметки.
- `canopy-stage-images --row 3 --phase leaf` — отдельные фото в стейджинг фенофазы (без cv2).

Предразметка, приём и обучение (реализованы зеркально cam1):
- `sam2_canopy` — авто-ПРЕДразметка слоя ОРГАНА: маска SAM2 → **полигон** YOLO-seg
  (а не бокс, как в cam1 `sam2_prelabel`). Класс органа выводится из фенофазы
  (`leaf`→leaf, `flowering`→inflorescence, `cluster`→cluster). Режимы: `auto`
  (SAM2AutomaticMaskGenerator — много листьев за раз, шумный черновик), `images`
  (промпты `boxes`/`points` из JSON или мышью). Это ЧЕРНОВИК слоя органа: слой
  ПОРАЖЕНИЯ и коды болезней доразмечаются руками. Выход → стейджинг фенофазы,
  совместим с `canopy-import`. severity тут не считается.
- `ingest canopy-import --labels <seg_export>` — YOLO-seg экспорт (полигоны) →
  `annotations/canopy_seg/{images,labels}` + `canopy_index.csv` (name,bush_code,phase).
  Картинки ищутся РЕКУРСИВНО (стейджинг вложен по фенофазе); идемпотентно; пустые .txt
  оставляются как healthy-негативы. Приём — чистый роутинг файлов, severity тут НЕ считается.
- `yolo_dataset --task canopy` — сборка YOLO-seg датасета (split по ряду, анти-утечка),
  классы — единый namespaced `CANOPY_SEG_CLASSES` (органы + поражения, `leaf_*`/`cluster_*`
  снимают коллизию `mildew`). `healthy` — не класс, а маска органа без слоя поражения.
- `train_canopy` — обучение сегментатора (Ultralytics `*-seg.pt`, ленивый импорт).

**Решение по геометрии классов:** один сегментатор органов+поражений (а не 4 задачи из
`CANOPY_LABELS.md §5` — это про инструмент разметки). severity = доля площади маски
поражения — считается из масок **в рантайме** (`CanopyAnalyzer` → `CanopyResult`), а не
при приёме датасета (там полигоны — просто метки).

**Незавершённый хвост cam2:** (1) рантайм `CanopyAnalyzer` — инференс сегментатора +
расчёт severity из площадей масок (shoelace) + сборка `CanopyResult`; (2) per-disease
уточнение классификатором на кропе органа (`§4`) — опционально, после пилота.

## Гайд по датасету (при работе с обучением)

Сплиты train/val/test режутся **по `bush_code` (лучше по `vine_row`)**, не по кадру:
кадры одного куста почти идентичны, кадр-уровневый сплит даёт утечку и завышенные метрики.
См. `DATASET.md` §«split по кусту».
